# Design-feedback 스킬

동료(PO·개발자·PM 등)가 만든 프로토타입에 대한 디자인 피드백을 "레벨 판정 → 범위 내 리뷰 → 태그 피드백 → 슬랙 회신 초안" 순서로 처리한다.

**기준 SSOT**: `~/agent-workspace/docs/design-feedback-criteria.md`
Step 1 전에 반드시 Read. 파일이 없으면 `references/criteria-snapshot.md` 사용 (스냅샷일 뿐, 충돌 시 SSOT가 우선).

**판단 근거 SSOT**: 디자인 아틀라스 (`Ohouse-product-design/ohouse-design/knowledge/`). 세션에 ohouse-design MCP가 있으면 그걸로 로드(list/get_design_knowledge, get_screen), 없으면 gh api로 raw 파일을 읽는다. 레벨별 로드 범위는 Step 2 참조.

---

## Step 0 — 인풋 수집

받을 수 있는 인풋과 처리:

| 인풋 | 처리 |
|---|---|
| 슬랙 스레드/메시지 링크 | Slack MCP(slack_read_thread 등)로 요청 내용 읽기 |
| 요청 텍스트 (복붙) | 그대로 사용 |
| Figma URL | Step 2에서 get_screenshot + 필요 시 get_design_context |
| 화면 이미지 / 프로토타입 링크 | 이미지는 직접 분석, 링크는 접근 가능하면 열람 |

요청에서 **판정 3요소**를 추출한다: ① 누가 쓰나 (오디언스) ② 몇 명이 얼마나 자주 쓰나 (규모) ③ 오류·실수 시 피해 유형 (돈 틀어짐 / 데이터 손상 / 잘못된 정보 고객 노출 / 없음 = 되돌리기 비용).
빠진 항목은 요청문·화면에서 추론하되, 출력에 `(추론)` 마커를 붙인다. 셋 다 추론 불가면 디어에게 그 항목만 짧게 확인.

요청자가 양식 없이 보낸 경우, 회신 초안 마지막에 다음부터 쓸 요청 양식(`references/request-template.md`)을 첨부할지 디어에게 제안.

## Step 1 — 레벨 판정

SSOT 문서의 4축(노출, 규모, 되돌리기 비용, 수명)으로 레벨을 정하고, 오디언스 보정을 적용한다.

- 셀러/파트너: 트래픽 낮아도 돈(정산·광고비·수수료)·데이터 걸리면 한 레벨 승격
- 브랜드사: 사용 빈도보다 관계 리스크로 판정
- 어드민: 새 인터랙션 패턴의 선례가 되는 화면이면 레벨 승격

출력 포맷:

```
[레벨 판정]
레벨: L2 (흐름 + 표준)
근거: 노출=셀러 / 규모=일 n회 운영 툴 / 되돌리기=정산 데이터 직결 → 승격 / 수명=지속
오디언스 관점: 셀러 — 정보 정확성, 반복 작업 효율에 힘줌. 비주얼 폴리시는 안 봄
이번 리뷰에서 보는 것: …
안 보는 것: …
```

판정 후 디어 컨펌 받고 Step 2 진행. 디어가 레벨을 조정하면 그 레벨로.

**L3 판정 시**: 리뷰를 진행하지 않는다. "이건 리뷰 범위가 아니라 디자이너가 협업자로 들어가야 하는 건"이라고 안내하고, design-review 스킬 워크플로우(페르소나 → 유저보이스 → 리뷰 → 스펙 → 크리틱)로 전환할지 확인.

## Step 2 — 범위 내 리뷰 (판단 기준 12개로만 판단)

판정된 레벨의 범위만 본다. 레벨 밖 이슈가 보여도 본 리뷰에 섞지 않는다 (Step 3의 "범위 밖 메모" 한 줄로만).

**순회 인덱스는 SSOT §4의 판단 기준 12개, 판단 권위는 디자인 아틀라스.** 모든 지적은 아틀라스 출처를 인용해야 한다 (최대 2개). 인용할 출처가 없으면 취향이므로 [사소]로 강등하거나 뺀다.

| 레벨 | 순회하는 기준 | 아틀라스 로드 |
|---|---|---|
| L1 | 1~4 (안전선: 오해 방지, 비가역 보호, 정보 신뢰, 패턴 재사용) | 로드 없이 진행 가능 (인용은 SSOT §4 출처 컬럼) |
| L2 | 1~9 (+ 과업 완결, 상태 완결, 일관성, 실수 방지·효율, 카피 명확성) | feedback-format + principles 로드. 카피 이슈 시 ux-writing |
| L2+ (AI 프로토타입) | + 13 비주얼 베이스라인 (강조 과잉, 색 남발, 토큰 이탈, 장식 노이즈, 브랜드 톤, 타이포 기본 = 바이브코딩 티 제거. 오디언스 무관 적용) | + visual-design 로드 |
| L3 | 1~12 — 단 L3는 design-review로 전환이 원칙 | design-review 워크플로우가 담당 |

**Screen 정책 확인 (고객단 화면, L2 이상)**: 크리틱 전 이 화면이 어느 Track Screen인지 식별하고 ohouse-design MCP get_screen으로 정책을 확인한다. Screen 정책이 최우선 인용 권위. 화면 식별이 안 되면 첫 질문은 "이 화면이 어디인가요?".

**시작 5질문 (L2 이상, feedback-format §1-1)**: 문제→솔루션 적합성 / 변수 통제 / 디폴트값 / 화면 1순위 / 일관성. 흐릿한 1~2개는 지적이 아니라 질문으로 되돌려준다.

**스코프 주의**: principles.md는 B2C 전용. 어드민 화면에는 B2C 규칙(빈 상태 일러스트, 하단 sticky CTA, 친근 톤)을 그대로 적용하지 않는다. 어드민은 `tracks/admin/_common/` 가이드 우선.

**오디언스별 우선 순회** (SSOT §4 표): 일반 유저 = 9·11 가중 / 셀러·파트너 = 3·8 우선, 11 제외 / 브랜드사 = 3·11(신뢰) / 어드민 = 1~4 + 8.

**보조 스킬 연결**
- 기준 6(상태 완결)을 깊게 점검해야 하면 state-verifier 스킬 병용
- 기준 9(카피 명확성): 인용 출처는 아틀라스 ux-writing.md, 대안 문구 작성은 **writing-bot 스킬**
- 기준 7(일관성)의 컴포넌트 확인은 ODS MCP(ods-prototype) 활용 가능. ODS 미사용·Track 미등록은 [제안] 이상, 디자인 토큰 위반은 [블로커] (feedback-format §5-2)

## Step 3 — 태그 피드백 출력

모든 항목에 태그 + 아틀라스 출처(최대 2개)를 단다. 태그 순서대로 정렬 (블로커 먼저). 크리틱 작법은 feedback-format을 따른다: 결론 먼저, 우려 최대 4개, 40줄 미만, 체크리스트 줄줄 나열 금지.

```
[피드백 — L2 / 셀러 관점]
요약: {전반 인상 + 결론 한 문장}

[블로커] {위치}: {문제} (근거: 원칙 5 에러·빈 상태). 제안: {대안}
[제안] {위치}: {문제} (근거: design-insight #8). 제안: {대안}
[사소] {위치}: {참고. 주관 의견이면 명시}

질문 (0~3개): {yes/no로 답 가능한 확인 질문}
범위 밖 메모 (선택): {L3급 이슈가 보였다면 한 줄만. 없으면 생략}
```

- 태그 = 아틀라스 severity 매핑: [블로커] = 🔴 차단 / [제안] = 🟡 강한 지적 / [사소] = 🟢 가벼운 제안
- [블로커] = 반영 안 하면 내보내면 안 됨. **기준 1~4(안전선) 위반, 접근성(WCAG AA)·디자인 토큰 위반은 자동 블로커**
- [제안] = 반영 권장, 판단은 요청자. 기준 5~9 위반은 기본 [제안], 유저 피해가 크면 [블로커]로 승격
- [사소] = 참고만. 출처 인용 없는 코멘트는 여기까지만 허용
- 블로커가 0건이면 "블로커 없음, 내보내도 됨"을 명시 (요청자가 가장 알고 싶은 것)
- 우려끼리 충돌하면 원칙 우선순위 적용: 접근성 > 가독성 > 일관성 > 결정 피로

## Step 4 — 슬랙 회신 초안

디어가 원하면 회신 초안을 작성한다. 반드시 `~/.claude/memory/work/feedback/feedback_slack-message-tone.md`의 규칙과 디어 말투를 적용:

**본문 구조 (2026-07-23 디어 확정)**:
- **카테고리는 고정하지 않고 그 건의 피드백 내용에 맞게 분류**해서 `[대괄호]` 섹션으로 묶는다. 예: `[사용성]` `[디자인]` `[데이터 신뢰]` `[케이스 고려]` `[확장성]` `[정책 확인]`. 억지로 채우지 않는다 (해당 피드백 없으면 그 카테고리 생략).
- **불렛마다 우선순위를 붙인다.** Step 1~3 판정에서 도출: [블로커]=P0(발행·머지 불가), [제안] 중 유저 피해·회귀 근거가 강한 것=P1(중요), 일반 [제안]=P2(권장), [사소]=P3(백로그·참고). 첫 등장에만 한글 설명 병기.
- **우선순위는 백틱 인라인 코드로 감싼다** (`` `P0` `` 형태). 슬랙에서 회색 배경 칩으로 렌더링돼 뱃지처럼 보인다.
- 카테고리 안에서 P0 먼저 정렬. P0이 있으면 요약 줄에서 "제일 중요한 건 ~"으로 먼저 짚는다.
- **한 불렛 = 피드백 하나.** 여러 지적을 한 불렛에 섞지 않는다.

- 첫 줄 감사 + 시그니처 emoji (`:cat_pray:` 등), 긍정 평가 한 줄 먼저 ("전체 흐름은 좋아요")
- 마커 라벨은 직관적 한국어: `[이것만 고치면 돼요]` `[제안]` `[참고]`
- 질문·요청마다 부담 완충 ("판단은 편하신 대로!", "없으면 패스!")
- 레벨·범위 선언을 자연어로 한 줄 ("어드민 화면이라 안전선 위주로 봤어요")
- 긴 줄표 사용 금지, 범용 유니코드 이모지 금지

**전송은 초안까지만.** 디어 확인 후에만 Slack MCP로 전송한다.

## Step 4.5 — 에이전트 수정 프롬프트 (옵션)

요청자가 바이브코딩 도구로 직접 수정할 수 있게, 피드백을 실행 가능한 프롬프트로 번역해서 슬랙 회신에 첨부 제안한다.

규칙:
- **프롬프트는 1개**, 내부는 우선순위(P0→P3) 순 체크리스트
- 에이전트 작업 지시: "위에서부터 한 항목씩 수정, 항목마다 결과 확인 후 다음 진행" (한 번에 갈아엎기 방지)
- **스코프 가드 필수**: "체크리스트에 없는 것(레이아웃 구조, 섹션 순서, 카피)은 바꾸지 마"
- **사람 입력 필요 항목은 에이전트가 값을 지어내지 못하게 분리**: 실데이터·색값·정책 답변은 기존 값 유지 + `TODO:` 주석 처리 지시. 프롬프트 말미에 "요청자가 채울 것" 목록으로 명시
- 항목마다 완료 기준(검증 가능한 상태)을 한 줄로: "390px에서 가로 스크롤 없음"처럼
- P3는 체크리스트 말미에 "여유 있으면" 섹션으로 (백로그 성격 유지)

## Step 5 — 사례 기록 (경계 보정용)

리뷰 완료 후 `~/agent-workspace/docs/design-feedback-cases.md`에 한 줄 append:

```
| 날짜 | 화면 | 오디언스 | 판정 레벨 | 블로커 수 | 경계 애매 여부 + 메모 |
```

사례 3~5건 쌓이면 L1/L2 경계 보정을 디어에게 제안. 파일 없으면 헤더와 함께 생성.
