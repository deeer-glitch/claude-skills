# 디자인 피드백 레벨 기준 (v1, 개인용)

> 2026-07-22 작성. 동료(주로 PO, 개발자·PM 포함)가 만든 프로토타입에 디자인 피드백을 줄 때, 어디서부터 어디까지 볼지 판정하는 기준.
> 일단 디어 개인 기준으로 운영. 팀 공통 규칙으로 확장 시 "L3 = 리뷰가 아니라 협업 전환" 부분을 PO 등 메이커 조직과 합의 필요.

---

## 1. 판정: 4가지 리스크 질문

레벨을 정하기 전에 아래 4개를 확인한다. 고객단/어드민, 트래픽은 대리 지표일 뿐이고, 실제 기준은 리스크.

| 축 | 질문 | 높음의 예 |
|---|---|---|
| 노출 | 누가 보는가? | 고객 > 파트너/판매자 > 사내 어드민 |
| 규모 | 얼마나 많이, 자주 쓰는가? | 홈·PDP급 트래픽, 매일 쓰는 운영 툴 |
| 되돌리기 비용 | 잘못되면 무슨 일이 생기고, 고치기 쉬운가? | 결제·할인·정산·데이터 삭제, 새 패턴 선례 |
| 수명 | 일회성인가 계속 남는가? | 지속 기능 > 한시 이벤트 > 내부 실험 |

주의: 어드민이라도 새 인터랙션 패턴을 처음 만드는 화면이면 이후 어드민 전체의 선례가 되므로 레벨을 올린다.

축의 역할 분리: 위 4축은 **얼마나 깊게 볼지(레벨)** 를 정하고, 오디언스(§3)는 그 레벨 안에서 **무엇에 힘줄지(관점)** 를 정한다.

## 2. 피드백 3레벨

| | L1. 안전선 체크 | L2. 흐름 + 표준 | L3. 디자인 오너십 |
|---|---|---|---|
| 대상 | 사내 어드민 저사용, 내부 실험 | 어드민 고사용, 고객단 저트래픽, 파트너 툴 | 고객단 고트래픽, 돈/신뢰 관련, 새 패턴 선례 |
| 디자이너 역할 | 리뷰어 (코멘트만) | 리뷰어 + 수정 제안 | 오너 (설계에 직접 참여) |
| 타임박스 | 30분 내 비동기 코멘트 | 1시간 내외, 필요 시 짧은 싱크 | 기존 디자인 프로세스 그대로 |
| 보는 것 | 판단 기준 1~4 (안전선) | 판단 기준 1~9 (+ 플로우, 상태/엣지케이스, 디자인 시스템, 핵심 카피) | 판단 기준 1~12 전부 (+ IA, 페르소나/VOC 검증) |
| 안 보는 것 | 픽셀, 컬러 톤, 마이크로카피, 레이아웃 취향 | 픽셀 단위 디테일, 비주얼 폴리시 | 없음 |

L3 판정이면 리뷰로 처리하지 않는다. 프로토타입이 이미 있어도 디자이너가 협업자로 들어간다.

## 3. 오디언스별 리뷰 관점

레벨은 깊이를 정하고, 오디언스는 같은 레벨 안에서 무엇에 힘줄지를 정한다.

| 오디언스 | 특성 | 레벨 판정 시 주의 | 힘줄 것 | 힘 뺄 것 |
|---|---|---|---|---|
| 일반 유저 | 비전문, 대체 서비스로 이탈 쉬움 | 트래픽 기준 그대로 적용 | 첫인상, 이해 용이성, 브랜드 톤, 비주얼 디테일 | (L3면 없음) |
| 셀러/파트너 | 전문 반복 사용자, 돈 직결 (정산·광고비·수수료) | **트래픽 낮아도 돈·데이터 걸리면 한 레벨 올림** | 정보 정확성, 반복 작업 효율, 오류·실수 방지 | 비주얼 디테일·브랜드 톤 다듬기 |
| 브랜드사 | 소수 담당자, 관계 임팩트 큼, 온보딩 가능 | 사용 빈도보다 관계 리스크로 판정 | 신뢰감·전문성 인상, 데이터 정확성 | 마이크로 인터랙션 |
| 사내 어드민 | 내부, 교육으로 커버 가능 | 기존 기준 그대로 | 안전선, 비가역 액션 | 그 외 전부 (L1 기준) |

## 4. 판단 기준 12개 (피드백의 근거)

**판단 근거의 SSOT는 디자인 아틀라스**(`Ohouse-product-design/ohouse-design/knowledge/`, ohouse-design MCP로 로드). 아래 기준 12개는 "무엇을 볼지" 순회 인덱스이고, "왜 문제인지"의 권위는 아틀라스 출처를 인용한다.

**모든 피드백 코멘트는 아틀라스 출처를 인용한다 (최대 2개, feedback-format §6). 인용할 출처가 없으면 취향이므로 [사소]로 강등하거나 뺀다 (주관 의견임을 명시).**

코멘트 형식: `[태그] {위치}: {문제} (근거: {출처 1~2개}). 제안: {대안}`

인용 우선순위 (feedback-format §6): ① Track Screen 정책 ② design-insight #N ③ principles 원칙 #N ④ ux-design/ux-writing/visual-design ⑤ 학계·외부 표준.

### 안전선 = 기준 1~4 (전 레벨, 위반 시 자동 [블로커])

| # | 기준 | 체크 질문 | 1차 출처 (아틀라스) |
|---|---|---|---|
| 1 | 오해 방지 | 유저가 라벨·문구를 보고 실제와 다른 결과를 기대하게 되지 않는가 | ux-writing / design-insight #11 |
| 2 | 비가역 보호 | 되돌릴 수 없는 액션(삭제·발송·확정) 앞에 확인 또는 복구 장치가 있는가 | 학계 (Nielsen #5 오류 예방) |
| 3 | 정보 신뢰 | 금액·날짜·수치가 정확한가. 불확실한 값이 확정처럼 보이지 않는가 | ux-writing (사실 + 행동 제안) |
| 4 | 패턴 재사용 | 같은 문제를 푸는 기존 컴포넌트·패턴이 있는데 새로 만들지 않았는가 | 원칙 9 Composition Order / feedback-format §5-2 |

### L2 추가 = 기준 5~9 (기본 [제안], 유저 피해가 크면 [블로커])

| # | 기준 | 체크 질문 | 1차 출처 (아틀라스) |
|---|---|---|---|
| 5 | 과업 완결 | 진입부터 완료까지 막힘 없이 가는가. 완료 후 다음 행동이 이어지는가. 취소·뒤로가기 경로가 있는가 | ux-design / 원칙 2 결정 피로 |
| 6 | 상태 완결 | empty·로딩·에러·권한 없음·극단값(0건, 999건, 긴 텍스트)이 정의됐는가 | 원칙 5 에러·빈 상태 퍼스트 시민 / 원칙 6 로딩 |
| 7 | 일관성 | ODS 컴포넌트를 쓰는가. 같은 동작이 서비스 다른 곳과 같은 방식으로 표현되는가 | 원칙 9 / feedback-format §5-2 (토큰 위반은 차단급) |
| 8 | 실수 방지·효율 | 자주 하는 작업의 단계 수가 최소인가. 실수하기 쉬운 입력에 가드(기본값·포맷·검증)가 있는가 | design-insight #8 디폴트값 |
| 9 | 카피 명확성 | 버튼이 결과를 말하는가("확인" 대신 "삭제하기"). 용어가 서비스 표준 용어인가 | ux-writing (인용) + **writing-bot 스킬 (대안 작성)** |

### AI 프로토타입 추가 = 기준 13 비주얼 베이스라인 (L2부터, 바이브코딩 티 제거)

AI가 생성한 프로토타입이면 L2 이상에서 항상 순회한다. 비주얼 "폴리시"가 아니라 **브랜드 정합의 최소선**이므로 셀러/어드민 오디언스에서도 제외하지 않는다. 출처는 전부 visual-design.md.

| 체크 | 질문 | 근거 |
|---|---|---|
| 강조 과잉 | 하이라이트·볼드·컬러 강조가 화면당 1~2개를 넘지 않는가 | design-insight #3 (정보 1~2개만 강조) |
| 색 남발 | 강조색이 1개로 통제되는가. 흔한 블루를 또 쓰지 않았는가. 배경색으로 강조하지 않았는가 | design-insight #1 (색 절제, 배경색은 강조의 적) |
| 토큰 이탈 | 오늘의집 토큰 밖 하드코드 팔레트(시안·네이비 계열 등 전형적 AI 팔레트)를 쓰지 않았는가 | visual-design 컬러 시스템 (하드코드 hex 금지) |
| 장식 노이즈 | 그라데이션·과한 그림자·장식 문자(◆ 등)·의미 없는 일러스트가 없는가. 구분은 박스보다 여백인가 | Tufte Chartjunk / design-insight #4 |
| 브랜드 톤 | 따뜻함·정돈됨과 맞는가. 제네릭 SaaS 랜딩 문법(영문 아이브로우 라벨, 통계 카드 3분할 등)을 그대로 쓰지 않았는가 | visual-design 브랜드 정체성 |
| 타이포 기본 | keep-all, 행간 1.5+, Pretendard, 영문 스케일 미적용인가 | 원칙 3 한글 가독성 |

| # | 기준 | 체크 질문 | 1차 출처 (아틀라스) |

### 디자인 아틀라스 연계 규칙 (2026-07-22)

- **크리틱 작법은 feedback-format.md를 따른다**: 우려 최대 4개, 인용 최대 2개, 결론 먼저, hedged 톤(단정·심판조 금지), 40줄 미만, 체크리스트 줄줄 나열 금지
- **태그 = severity 매핑**: [블로커] = 🔴 차단 / [제안] = 🟡 강한 지적 / [사소] = 🟢 가벼운 제안. 팀 크리틱 문화와 호환
- **원칙 충돌 시 우선순위** (principles ⚖️): 접근성 > 가독성 > 일관성 > 결정 피로 > 나머지
- **스코프 주의**: principles.md는 B2C 전용. 어드민 화면에는 B2C 규칙(빈 상태 일러스트, 하단 sticky CTA, 친근 톤)을 그대로 적용하지 않는다. 어드민은 `tracks/admin/_common/` 가이드 우선
- **고객단 화면은 크리틱 전 Track Screen 정책 확인** (ohouse-design MCP get_screen). Screen 정책이 최우선 인용 권위
- L2 이상에서는 시작 전 feedback-format §1-1의 5질문(문제→솔루션 적합성, 변수 통제, 디폴트값, 화면 1순위, 일관성) 중 흐릿한 1~2개를 질문으로 되돌려준다

|---|---|---|
| 일반 유저 | 9, 11 가중 | 없음 |
| 셀러/파트너 | 3, 8 우선 | 11 제외 |

| 사내 어드민 | 1~4 + 8 | 나머지 후순위 |

## 5. 운영 규칙

### 피드백 태그
모든 코멘트에 셋 중 하나를 단다.

| 태그 | 의미 |
|---|---|
| `[블로커]` | 반영 안 하면 내보내면 안 됨 |
| `[제안]` | 반영 권장, 판단은 요청자 |
| `[사소]` | 참고만 |

### 리뷰 요청 시 요청자가 적을 3줄
1. 누가 쓰나 (일반 유저 / 셀러·파트너 / 브랜드사 / 사내 어드민)
2. 몇 명이, 얼마나 자주 쓰나 (예: 운영 담당 3명이 주 1회 / 셀러 수천 명이 매일 / 전체 고객 트래픽)
3. 오류나 실수가 나면 무엇이 잘못되나 (돈이 틀어짐: 결제·정산·할인 / 데이터가 지워지거나 바뀜 / 잘못된 정보가 고객에게 보임 / 없음: 조회용)

이 3줄로 레벨을 판정하고, 레벨 협상도 이 기준으로 한다.
