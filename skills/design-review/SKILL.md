## 페르소나 컨텍스트

> 검증 시 기준이 되는 페르소나는 **서비스를 실제로 사용하는 일반 사용자**입니다.
> 출처: 오늘의집 인사이드아웃 (내부 리서치 문서, 2024.08 기준)

### 페르소나 분류 체계

오늘의집 고객은 **관심사 / 탐색 방식 / 생산자** 3축으로 총 10개 그룹으로 분류됩니다.
핵심 그룹(MAU 43%)이 전체 GMV의 98%를 차지합니다.

### [Interest] 관심사 그룹

| 페르소나 | 핵심 욕구 | MAU | 주요 특성 |
|---------|----------|-----|----------|
| **I-1 이사** | 내 취향에 맞는 가구/가전을 사고 싶어 | 26% (92만) | 가전 기능 학습 어려움, 가구는 스타일 조화를 중시. 이사 시기가 구매에 직결 |
| **I-2 스타일링** | 집을 예쁘게 꾸밀 아이템을 찾고 싶어 | 17% (61만) | 콘텐츠 소비량 이사 그룹보다 25% 높음. 다른 사람의 스타일링 참고 빈도 높음 |
| **I-3 요리** | 쿡웨어와 식자재로 요리 생활을 즐기고 싶어 | 10.5% (37만) | 89%가 주 1회 이상 요리. 오늘의집에서 요리 콘텐츠 찾기 어렵다고 인식 |
| **I-4 살림수납** | 살림 꿀템으로 살림을 잘하고 싶어 | 7.4% (26만) | 인당 구매 건수 월 2.02건. 쿠팡보다 오늘의집 먼저 탐색 |
| **I-5 바이너리** | 취향을 넓혀 나만의 공간을 만들고 싶어 | 2.6% (9.5만) | GMV/인 1위(₩196,263). 콘텐츠 조회 월 118회 |

### [Exploring] 탐색 방식 그룹

| 페르소나 | 핵심 욕구 | MAU | 주요 특성 |
|---------|----------|-----|----------|
| **E-1 HCPY** | 콘텐츠 보다 보니 지갑을 열게 돼 | 5.4% (19만) | GMV/인 ₩313,692. 콘텐츠 → 상품 발견 → 후기 확인 → 구매 |
| **E-2 HCPN** | 콘텐츠는 많이 보지만 아직 구매는 못 해 | 8.2% (29만) | 이사 전 탐색 중. O2O 신청 비율 HCPY 대비 2.9배 |
| **E-3 LCPY** | 이미 외부에서 결정했어, 여기선 구매만 | 11.5% (41만) | 리텐션 가장 낮음(M3 54%). 주문 후 1분 내 앱 이탈 |

### [Creating] 생산자 그룹

| 페르소나 | 핵심 욕구 | MAU | 주요 특성 |
|---------|----------|-----|----------|
| **C-1 리뷰 작성자** | 경험을 공유하고 포인트도 받고 싶어 | 2.3% (8.2만) | 이타적 동기와 보상 동기 공존 |
| **C-2 콘텐츠 업로더** | 내 취향과 일상 감각을 자랑하고 싶어 | 0.45% (1.6만) | 콘텐츠 조회량 전체 1위 |

### 기본 페르소나 (미지정 시 — E-2 HCPN)

| 항목 | 내용 |
|------|------|
| 그룹 | E-2 HCPN (콘텐츠 고소비 + 미구매) |
| 목표 | 이사 전 인테리어 레퍼런스 탐색 및 상품 리스트업 |
| 행동 | 월 콘텐츠 22회 이상 조회, 상품 1,200회 이상 조회하지만 구매 없음 |
| 미구매 이유 | 가격 부담, 원하는 상품 미발견, 아직 이사 시기 아님 |
| 디바이스 | 모바일 우선 |
| 특성 | O2O 신청 비율 HCPY 대비 2.9배. 집들이 콘텐츠 소비 비중 높음 |

### 핵심 인사이트

- **관심사 다양성 = 리텐션**: 관심사가 많을수록 방문일 증가
- **이사 타이밍**: HCPN이 HCPY로 전환되지 못하는 가장 큰 이유는 '아직 이사 전'
- **콘텐츠 → 구매 연결**: 관심 카테고리 콘텐츠를 찾기 쉬우면 오늘의집에서 먼저 구매
- **LCPY 이탈**: 구매 후 1분 내 이탈. 주문완료 페이지가 이탈 방어 핵심 접점
- **미구매 주요 이유**: 가격(40.5%), 상품 미발견(16.4%)

## 워크플로 순서

### STEP 1 — #페르소나 (페르소나 워크스루)

위 페르소나 데이터 기반으로 시나리오 단계별 시뮬레이션.
타겟 유저 정의 → Step별 반응 및 예상 행동 → 이탈 위험 지점 출력.

출력 형식:
```
[페르소나 워크스루]
페르소나: {이름 및 특성 요약}
시나리오: {사용 맥락}

Step 1 — {화면/액션}: {페르소나의 반응 및 예상 행동}
Step 2 — ...

이탈 위험 지점: {단계명} — {이유}
감정 곡선 요약: {긍정 → 마찰 → 회복 흐름}
```

### STEP 1.5 — #유저보이스 (VOC 조회)

user-voice 스킬 실행. STEP 2 근거로 활용.
관련 키워드로 유대시 VOC 조회 후 정리.

출력 형식:
```
## 유저 Pain Point (출처: 유대시 VOC)

[키워드명] 관련 N건 의견 확인 (조회 기간: YYYY.MM ~ YYYY.MM)

주요 Pain Point:
1. [패턴화된 불만 요약] — 실제 인용: "..."
2. [패턴화된 요청 요약] — 실제 인용: "..."
```

---

### STEP 2 — #리뷰 (디자인 리뷰)

Nielsen 휴리스틱 평가 + 근거 기반 피드백 적용.
STEP 1.5 VOC를 근거로 활용.
잘된 부분과 짚을 부분(수정 필요 / 확인 필요) 분리 제공.
피드백 순서: 문제 지적 → 근거 → 대안 제시.

휴리스틱 평가 우선 기준:
1. 시스템 상태 가시성
2. 오류 방지 및 복구
3. 일관성 및 표준 준수
4. 인식 부하 최소화

---

### STEP 3 — 반영 확인

수정된 버전 재첨부 시 이전 리뷰 포인트 기준으로 항목별 체크.

출력: ✅ 반영됨 / ❌ 미반영 / 🆕 새로 짚을 부분

---

### STEP 4 — #스펙 (스펙 문서 작성)

확정된 디자인 기준으로 케이스별 상세 스펙 문서화.

출력 포맷:
- 번호 + 항목명: 내용 (플랫 리스트)
- 케이스 헤더: 조건 기준 [댓글 3개 이상] 형식
- 공통 스펙은 상단에 한 번만 분리

---

### STEP 5 — #크리틱 (디자인 크리틱 채널 초안)

`#des_pd_design_critique` 채널의 실제 관례 포맷으로 작성한다 (2026-06~07 포스트 30건 분석 기준).

출력 포맷:
```
*[진행상황 공유 | 파이널 공유 | 결과 공유] {과업명}* @{리뷰어} cc. @{관련자}
> [Figma {링크}]
> [PRD {링크}]  (없으면 "PRD 없음({사유})" 명시)

`배경`
- {비즈니스·데이터 맥락}

`문제`
- {유저 관점 문제 정의}

`핵심 가설`
- {~하면, ~할 것이다. 조건 1개 + 기대 결과 1개}

`목적/목표`
- {개선 목표}

`주요 개선 항목` (또는 `해결방향`)
1. {솔루션 1}
2. {솔루션 2}

`디자인 고려사항` (해당 시)
- {결정 + 결정 배경 한 줄}

`크리틱 요청 포인트` (3개 이내)
- {의견 요청 1}
- {의견 요청 2}
```

작성 규칙:
- 헤더 태그로 글의 단계를 선언: 진행상황 공유(피드백 요청) / 파이널 공유(확정) / 결과 공유(실험 결과)
- 섹션 라벨은 백틱 또는 볼드, 한 글에서 한 방식으로 통일. 불렛 위계 • → ◦ → ▪︎, 번호는 선택지 나열(1안/2안)에만
- 명사형 종결 기본("~함", "~됨", "~예정"). "~습니다"와 혼용 금지
- 수치는 기간·모수 병기("모달 노출 10.7만 건 중 약 65%"), UT·VOC는 원문 인용
- 결정에는 배경 한 줄("~로 판단해 A안 채택"). 미확정 지점은 "현재 고민 중"으로 열어 크리틱 요청 포인트로 연결. 반영한 피드백은 "(크리틱 의견 반영)" 표기
- em 대시(—) 금지. 미확정 정책·금액은 "확정 전"/"가안" 명시

## 운용 원칙

- 각 단계 완료 후 다음 단계 진행 여부를 확인
- 노션 문서는 명시적 OK 후에만 수정
- 불확실한 내용은 "~일 수 있습니다"로 표기
- 확신도 90% 미만인 판단은 반드시 불확실성 표기
- 비즈니스 지표 연결 불가 시 "데이터 부족으로 추정 어려움" 명시
- 팀 전체 스킬이므로 특정 개인 표현 금지
