# Masking Rules — 공유 모드에서 자동 적용

공유 모드 발동(PD OK) 시 푸시 직전 자동 마스킹. 3중 안전장치 거침. 이 문서는 *어떻게* 마스킹하는지 정의.

## 3중 안전장치

1. **자동 룰** — 이 문서 + `assets/masking-patterns.json` 기반. `scripts/mask.py`가 실행.
2. **수동 추가** — PD가 *"이것도 가려"* 발화 시 추가 패턴 적용.
3. **푸시 전 미리보기** — 마스킹된 HTML을 `~/Downloads/{filename}.masked.html`로 별도 저장. PD가 직접 열어 확인 후 *"OK 푸시"* 받은 후에야 푸시.

→ 자동 룰이 100% 잡는다고 믿지 말 것. 미리보기 단계가 결정타.

## 마스킹 카테고리별 룰

### 1. 사내 이름 → 역할명

`masking-patterns.json`의 `internal_names` 카테고리에 **본인 프로젝트 팀 구성원 이름**을 채워서 쓴다. 예:

| 원본 | 마스킹 |
|---|---|
| (팀 PO 이름) | PO |
| (팀 디자이너 이름) | PD |
| (팀 개발자 이름) | FE / BE |
| (팀 QA 이름) | QA |

→ 처음 스킬을 쓸 때 본인 프로젝트에서 자주 등장하는 이름을 정리해서 `masking-patterns.json`의 `internal_names.rules`에 추가해두면 이후 자동 마스킹된다. 등장 빈도 높은 이름 우선. 정규식 매칭 + 한글/영문 둘 다.

### 2. 사내 이메일 → 완전 제거

- `*@bucketplace.net` → 통째 제거
- `*@dailyhou.se` → 통째 제거

### 3. 사내 도메인 URL → 제거 또는 `[내부 링크]`

| 도메인 | 처리 |
|---|---|
| `*.bucketplace.net` | 제거 |
| `xpc.dailyhou.se` | 제거 |
| `redash-commerce.datahou.se` | 제거 |
| `*.datahou.se` | 제거 |
| `notion.so/ohouse/*` | 제거 (또는 `[내부 문서]`) |
| `ohou-se.slack.com/*` | 제거 |
| `ohouse.atlassian.net/*` | 제거 |
| `figma.com/design/*` (사내 파일) | 제거 |
| `docs.google.com/*` (사내 시트·문서) | 제거 |
| `prod-files-secure.s3.us-west-2.amazonaws.com/*` (Notion 첨부) | 제거 |

→ 헤더의 문서 링크 칩(`.doc-chip`)도 *사내 URL이면 통째로 칩 자체 제거*. 외부 공개 URL은 유지.

### 4. Jira 키 / 실험키 → `[티켓]` / `[실험키]` 또는 제거

| 패턴 | 마스킹 |
|---|---|
| `COMMPL-\d+` | `[티켓]` |
| `COMMPO-\d+` | `[티켓]` |
| `OHSAND-\d+`, `OHSIOS-\d+`, `COMMWEB-\d+`, `CONTWEB-\d+` 등 사내 Jira 키 | `[티켓]` |
| `XPC \d+`, `XPC1\d{3,4}` | `[실험키]` |

### 5. 실측 데이터 → 제거 또는 `+N%` 일반화

| 패턴 | 처리 |
|---|---|
| `user CTR \d+(\.\d+)?%` | `user CTR +N%` 또는 제거 |
| `GMV [+-]\d+(\.\d+)?%` | `GMV ±N%` |
| 구체 표본 크기 (`733개 상품`, `17.1만 조회` 등) | 제거 또는 `[N개]` |
| 구체 거래액·인원 수 | 제거 |

### 6. 실제 가격 → 더미 또는 placeholder

| 원본 | 마스킹 |
|---|---|
| 캡처 인용 정확한 금액 (예: `5,316,500원`) | `XXX,XXX원` 또는 보편 더미 (`890,000원` 같은 둥근 수) |
| 더미 가격 (`89,000원`, `59,000원` 등 이미 더미인 것) | 유지 |

→ *캡처 인용 가격*과 *원래 더미 가격*을 구분. 캡처 인용은 footnote에서 *"실측"* 모드였을 가능성 → 더미로 강제. 본인 프로젝트에서 자주 캡처하는 금액이 있으면 `masking-patterns.json`의 `real_prices_from_capture`에 규칙을 추가해둔다.

### 7. 사내 약어 / 명칭 → 일반화

본인 프로젝트의 내부 코드네임·모듈 약어(예: 사내에서만 쓰는 지면 번호, 기능 줄임말)를 `masking-patterns.json`의 `internal_codenames`에 추가해서 일반화한다.

| 유형 | 처리 예시 |
|---|---|
| 내부 지면/모듈 번호 | 일반 명칭으로 (예: "지면4" → "추천 모듈") |
| 기능 줄임말 | 풀어서 표현 (예: "브묶할" → "브랜드 묶음 할인") |
| 사내 프로젝트 코드명 | 제거 또는 "본 과업"으로 일반화 |
| 반기 코드 (`25H1`, `26H2` 등) | 제거 (기본 포함) |

→ *공개 기능명*(오늘의집 공식 서비스명)은 유지. *내부 코드네임·약어*만 일반화.

### 8. PRD 직접 인용 → 일반화

| 원본 | 마스킹 |
|---|---|
| "PRD U-2 정의대로면..." | "기존 정책 기준으로는..." |
| "PRD v0.2 (작성자명, 날짜)" | "내부 문서 (v0.2)" |
| 사내 프로젝트 코드가 포함된 문구 (예: "26H1_○○Growth") | 제거 |

### 9. KV 사회적 증거 (실데이터 노출) → 더미 강제

KV에 `so******님의 패키지 5,316,500원` 같은 *실제 유저 데이터* 형식이 있을 경우:
- ID는 이미 마스킹된 형태(`so******`)라도 → 완전 더미 ID로 (`u****님`)
- 금액 → 더미 강제
- 카테고리 구성 → 일반 묘사

## 마스킹 안 함 (예외 — 공개 가능)

| 항목 | 이유 |
|---|---|
| `오늘의집` 브랜드명 | 공개 |
| 일반 가구·가전 브랜드명 (삼성, LG, 한샘, 시디즈 등) — 공개 매장에서 쓰는 표현 수준이면 | 공개 정보 |
| `Pretendard`, `Apple SD Gothic Neo` 같은 외부 폰트명 | 공개 |
| 일반 카테고리명 (침대, 소파, 매트리스 등) | 공개 |
| 더미 가격·카피·로고 (이미 더미라고 표기된 것) | 유지 |
| ODS 같은 공개된 디자인 시스템명 | 공개 (단 사내 Figma URL은 제거) |
| 오늘의집이 대외 공개한 서비스·기능명 | 공개 |

→ 단 *PD가 추가로 "이것도 가려" 발화*하면 그 항목도 마스킹.

## 푸시 직전 footnote 자동 추가

마스킹 처리 후 와이어 하단 footnote에 자동 추가:

> *공유용 — 민감 정보 마스킹됨. 카피·색·데이터 더미. 내부 SSOT와 다를 수 있음.*

## 정규식 매핑 (`assets/masking-patterns.json`)

`scripts/mask.py`가 읽는 패턴 사전. 이 문서의 규칙을 정규식으로 박은 것. 새 패턴 발견 시 JSON 업데이트 + 이 문서 업데이트 둘 다.

**처음 쓰는 경우**: `internal_names` / `internal_metrics` / `real_prices_from_capture` / `internal_codenames` 카테고리는 프로젝트마다 다르므로, 본인이 자주 다루는 팀 구성원·지표·코드네임으로 한 번 채워두는 걸 권장. `internal_emails` / `internal_urls` / `jira_experiment_keys`는 회사 공통이라 그대로 써도 됨.
