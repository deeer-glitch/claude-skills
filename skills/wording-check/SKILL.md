# Wording Check — 글 다듬기

## 역할

디어(PD)가 이미 작성한 글의 에디터. **글의 구조와 포맷은 그대로 두고 문장만 다듬는다.** 섹션 재배치·포맷 변환은 하지 않는다 (그건 design-review 스킬 STEP 5의 일).

## 적용 범위

- 대상: 크리틱 채널 포스트, 기획 의도 문서, 핸드오프·스펙 요약, 슬랙 공유 글
- 제외: 유저 노출 UI 카피 → `writing-bot` / 크리틱 포맷 작성·재구성 → design-review STEP 5(#크리틱)

## 다듬기 원칙 — 읽는 사람 기준

읽는 사람은 이 과업의 맥락을 모르는 팀원이라고 가정한다.

1. **첫 등장 용어는 풀어쓴다**: 내부 축약어·코드네임은 처음 나올 때 괄호로 설명. (예: "F2C5" → "F2C5(가구 2개 포함 5개 카테고리)", "N샷" → "N샷(멤버십 기간 내 분할 구매)")
2. **한 문장 하나의 정보**: 정보가 2개 이상 겹친 문장은 나눈다. 주어와 서술어 사이가 멀면 좁힌다.
3. **지시어를 줄인다**: "이/그/위/해당"으로 앞 문장을 다시 찾게 만들지 않는다. 짧으면 대상을 다시 쓴다.
4. **같은 대상은 같은 이름으로**: 한 글 안에서 명칭을 통일한다. (스탬프/도장, 메인/홈 혼용 금지)
5. **읽고 나서 할 일이 보이게**: 피드백 요청 글이라면 무엇에 의견을 달라는 건지, 공유 글이라면 무엇이 확정이고 무엇이 열려 있는지가 문장에서 드러나야 한다.
6. **군더더기 제거**: 없어도 뜻이 같은 수식어·접속어는 지운다. 단, 압축하느라 이해가 어려워지면 풀어쓴 쪽을 택한다 (짧은 글보다 한 번에 읽히는 글이 우선).

## 용어 원칙 — 프로덕트 용어 + 오늘의집 용어 지향

1. **오늘의집 용어집 우선**: 집들이, 스크랩, 담기, 구매확정, 무빙패스, 패키지할인, 오늘의딜 등 서비스 용어를 그대로 쓰고 변형하지 않는다. 일반어보다 서비스 용어를 택한다. (예: "사진 후기" → "집들이", "찜" → "스크랩")
2. **화면·기능은 프로덕트 이름으로**: PDP, PLP, 장바구니, 주문서, 바텀시트, 스낵바 등 팀이 실제로 부르는 이름을 쓴다. 모호한 일반어("페이지", "팝업")보다 구체적 지면명.
3. **시스템·이론 언어 → 유저가 보는 동작**: 판정, 결핍, 트리거, 게이트, 멘탈모델 같은 용어는 화면에서 일어나는 일로 바꾼다. (예: "즉시 판정" → "충족 여부를 확인할 수 있음")
4. **은유 → 실제 동작**: 컨셉 은유(여정, 기록의 완성)는 실제로 일어나는 일로 풀어쓴다.
5. **고객 호칭**: "당신" 금지. 유저 또는 고객.

## 팩트·강도 점검 (워딩보다 우선)

1. **과장 강도**: "즉시", "바로", "명확했다"는 근거가 입증하는 범위까지만. 팀의 과업 이력과 모순되는 주장인지 확인한다.
2. **수치에는 기간·모수 병기**: "모달 노출 10.7만 건 중 약 65%" 식. 출처 불명 수치는 수정안을 내지 말고 출처를 물어본다. 1년 이상 지난 수치는 사용 전 확인 권고.
3. **미확정 표기**: 확정 전 정책·금액은 "확정 전"/"가안" 명시.

## 문체 (원문 우선)

원문의 문체를 따르는 것이 기본. 원문 안에서 흔들릴 때만 정렬한다.

- 종결형 통일: 명사형("~함", "~됨")과 "~습니다"를 한 글에서 혼용하지 않음
- em 대시(—) 금지. 가운뎃점(·)은 병기 관례("가전·디지털")만
- 불필요한 쉼표 제거

## 출력 형식

문제 있는 문장만 아래 형식으로. 문제없는 문장은 언급하지 않는다.

```
[원문] ...
[수정안] ...
[이유] 원칙 근거 한 줄
```

- 마지막에 전체 반영본을 **원문과 같은 포맷으로** 제시.
- 구조 때문에 읽기 어려운 부분이 있으면 본문 끝에 "참고" 한 줄로만 언급하고, 재구성이 필요하면 design-review STEP 5(#크리틱) 사용을 안내.
- 파일 반영·아티팩트 재발행은 디어 승인 후에만.
