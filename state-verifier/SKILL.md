# State-verifier 스킬

Figma 화면 구조를 읽고 UI 상태 누락과 엣지케이스를 탐지하여 체크리스트를 생성하는 팀 공용 스킬.

---

## Step 0 — 첫 사용 여부 확인

아래 신호를 기반으로 판단하며, 물어보는 것은 최후 수단이다.

**가이드 없이 바로 Step 1 진행하는 경우**
- URL 또는 이미지를 바로 전달한 경우 → 사용법을 아는 사람이므로 바로 진행
- "다시", "이번엔", "이 화면도" 등 연속 작업 표현이 있는 경우 → 바로 진행

**가이드를 먼저 안내하는 경우**
- "처음이에요", "어떻게 써요", "사용법 알려줘" 등 명시적 표현 → 바로 안내
- "state-verifier 써줘", "상태 체크 해줘"처럼 URL/이미지 없이 스킬만 언급한 경우 → 아래 문장으로 한 번 확인

```
처음 사용하시나요? 처음이시라면 스킬 사용 가이드를 먼저 안내해드릴게요.
```

가이드가 필요한 경우 `references/user-guide.md`를 읽어서 안내한 뒤 Step 1로 진행.

---

## Step 1 — 인풋 확인 및 분기

Figma URL 또는 이미지와 함께 아래 맥락 정보가 있으면 같이 받는다.
없으면 Figma 구조만으로 진행하되, 탐지 정확도가 낮아질 수 있음을 안내한다.

```
함께 주시면 분석 품질이 올라가요 (선택)
• 이 화면의 핵심 정책이나 결정사항
• PRD / 1Pager 요약
• 이전 대화에서 확정된 내용
```

### A. 단일 프레임 URL
`get_design_context` 1회 실행 → Step 2로 바로 진행.

### B. 섹션 URL
1. `get_metadata`로 섹션 구조 읽기
2. 주요 화면 프레임 목록 표시:

```
이 섹션에 {N}개 화면이 있습니다.

1. [프레임명] — node-id: XXXX:XXXX
2. [프레임명] — node-id: XXXX:XXXX
...

전체 처리할까요, 특정 화면만 할까요?
전체면 "전체", 특정 화면이면 번호를 말씀해주세요.
```

3. 선택에 따라 해당 프레임 `get_design_context` 실행

### C. 이미지 인풋
이미지를 직접 분석해서 화면 구조 추론.
상태 프레임은 이미지에 없으면 탐지 불가 → 구조 기반 추론으로 대체.

### D. URL/이미지 없이 텍스트만
기능명과 주요 컴포넌트를 질문하여 수동으로 채움.

## Step 2 — 분석

### 상태 탐지 기준

아래 키워드로 레이어명 / 프레임명 / 컴포넌트명을 스캔한다.

| 상태 | 탐지 키워드 |
|---|---|
| Loading / Skeleton | skeleton, loading, shimmer, placeholder |
| Empty | empty, 비어있음, 결과 없음, 🌀 Empty |
| Error | error, 오류, 실패, COMMON__ERROR, 연결 문제 |
| Success | success, default (데이터 있는 기본 상태) |
| Disabled | disabled, 비활성 |

**누락 판단**: 위 상태 중 Figma에서 확인되지 않으면 "누락 의심"으로 표시.
단, 해당 화면 성격상 불필요한 상태(예: 스켈레톤이 없어도 되는 정적 화면)는 제외.

### 엣지케이스 추론 기준

화면 내 컴포넌트 타입을 기반으로 아래 매핑으로 엣지케이스를 자동 생성한다.

| 컴포넌트 타입 | 추론되는 엣지케이스 |
|---|---|
| TEXT 레이어 / 카드 타이틀 | 텍스트 최대 길이 초과 시 말줄임 / 줄바꿈 처리 |
| Thumbnail / Image | 이미지 로드 실패 → placeholder / 대체 이미지 |
| 리스트 / 그리드 | 결과 0건(Empty) / 1건(레이아웃) / 최대 N건(페이지네이션) |
| SearchField / Input | 포커스·블러 상태 / 유효성 실패 에러 / 공백 제출 / 특수문자 |
| Button / CTA | Disabled 상태 / Loading 상태 (중복 탭 방지) |
| Avatar | 이미지 로드 실패 → 기본 아이콘 |
| Badge / Chip | 텍스트 길이 극단값 |
| 네트워크 의존 화면 전반 | 응답 지연 / 타임아웃 / 재시도 정책 |

## Step 3 — 체크리스트 출력

아래 포맷으로 출력한다. 화면/페이지 단위로 그룹핑하여 스캔하기 쉽게 구성한다.

```
[State-verifier] {섹션명 또는 과업명}

확인된 케이스

{화면명 1}
✅ {상태명}
✅ {상태명}

{화면명 2}
✅ {상태명}
✅ {상태명}

---

빠진 케이스

{화면명 1}
❌ {누락 상태} — {이유 또는 트리거 조건}

{화면명 2}
❌ {누락 상태} — {이유 또는 트리거 조건}
❌ {엣지케이스} — {조건 및 처리 방식 미정}

---

이 중에 개발 리뷰 단계에서 꼭 필요한 것들만 추리면:
• {화면명}: {핵심 케이스}
• {화면명}: {핵심 케이스}
```

---

## 출력 규칙

- **✅ 확인된 케이스**: Figma에서 실제 확인된 상태 프레임 또는 컴포넌트. 화면별로 그룹핑
- **❌ 빠진 케이스**: Figma에서 확인되지 않았거나 정의가 필요한 상태 및 엣지케이스. 화면별로 그룹핑
- 화면이 여러 개인 경우 각 섹션 내에서 화면별로 나누어 출력
- 체크리스트 마지막에 **개발 리뷰 단계에서 꼭 필요한 것들만 추린 요약**을 항상 포함
- 해당 없는 항목은 출력하지 않음 (빈 섹션 제거)"

---

## 운용 원칙

- Step 0(첫 사용 확인)은 스킬 첫 실행 시에만. URL/이미지를 바로 주면 생략하고 Step 1 진행
- 섹션 전체 처리 시 프레임 목록 먼저 확인 후 사용자 선택 대기
- 전체 처리 시 2~3개씩 묶어 순차 생성
- 확인되지 않는 상태는 "누락 의심"으로 표시하되, 단정하지 않음
- 팀 전체 스킬이므로 특정 개인 표현 금지
- 노션 저장은 명시적 요청 시에만

---

## Step 4 — 이어받기 요약 (항상 실행)

체크리스트 출력 완료 후 반드시 아래 형식으로 요약을 출력한다.
다음 대화 시작 시 이 내용을 그대로 붙여넣으면 맥락이 이어진다.

```
[이어받기 요약]
• 작업 화면: {화면명 또는 섹션명}
• 확정된 케이스: {✅ 항목 요약}
• 보완 필요: {❌ 항목 중 우선순위 높은 것}
• 다음 단계: {Figma 보완 후 spec-policy-handoff로 주석 작성 권장 등}
```
